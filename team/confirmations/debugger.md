# Debugger 확인 보고서 — Phase 0

> **역할**: Teammate 2 — 코드 디버거 (Debugger)
> **단계**: Phase 0 — 코드 작성 전 스펙/아키텍처 기반 위험 사전 식별
> **기준 문서**: CLAUDE.md, SPEC.md (v0.2), TEAM.md

---

## (a) 위험 로직 사전 식별

### A-1. 주차/나이 계산 (LMP·EDD 기반)

**위험도: 높음**

- **이중 기준 불일치**: LMP로 입력 시 `EDD = LMP + 280일`이 표준이나, 실제 산모의 배란일 차이(±14일)로 주수 오차 최대 2주 발생 가능. 의사가 초음파로 수정한 EDD가 있을 경우 앱 내 LMP 기반 계산값과 충돌함.
- **입력 미제공 처리 경로 미정의**: LMP와 EDD 중 하나만 필수(SPEC 1.2.1)라고 되어 있으나, 어떤 것도 입력하지 않고 스킵할 경우의 fallback 경로가 명시되어 있지 않음.
- **주수 표현 방식**: `D+주차`가 의미하는 바가 "완성된 주"(예: 12주 3일 → 12주)인지 "반올림 주"인지 정의 부재. 의학 표준(완성된 주 사용)을 따라야 하나 SPEC에 명시 없음.
- **윤년 경계**: LMP가 2월 28일이고 윤년인 경우, `+ 280일` 계산 시 2월 29일 포함 여부에 따라 EDD가 1일 달라짐. `Calendar.current.date(byAdding:)` API 사용 시 자동 처리되지만, 수동 계산 로직 작성 시 버그 발생 가능.
- **시간대(TimeZone) 경계**: 자정 전후 앱 실행 시 "오늘"의 기준이 사용자 기기 시간대 vs 서버 UTC에 따라 주수가 달라질 수 있음. CloudKit 동기화 시 특히 위험.
- **다태아 주수**: 쌍둥이는 EDD가 단태아보다 3주 빠른 37주를 기준으로 하는 경우가 있으나, SPEC에 다태아 EDD 계산 차이 처리 정책이 없음.

### A-2. Pregnancy → Child 전환 로직

**위험도: 높음**

- **전환 트리거 타이밍 미정의**: "출산 예정일 전후로 전환 플로우 제공"(SPEC 1.3)이라 되어 있으나, 조산(37주 미만), 지연 출산(42주 이후), EDD를 4주 이상 넘긴 경우에 전환 플로우가 언제 등장하는지 로직이 명시되어 있지 않음.
- **데이터 승계 원자성(Atomicity) 미확보**: 배 사진 → 성장 사진 타임라인 전환, 태명 → 이름 전환 등 다수의 데이터를 동시에 이전하는 작업 중 앱 종료/네트워크 단절 시 반쪽짜리 전환(partial migration) 상태가 지속될 수 있음. CoreData 트랜잭션 + CloudKit 동기화 조합에서 롤백 보장이 핵심.
- **Pregnancy 보존 vs Child 생성 동시성**: 임신 기록을 "태아 시절 아카이브로 영구 보존"하면서 Child 엔티티를 새로 생성하는 이중 구조에서, 동일 데이터(배 사진 등)가 중복 참조될 경우 삭제 시 고아(orphan) 레코드 발생 위험.
- **다태아 전환**: 쌍둥이의 경우 Pregnancy 1개에서 Child 2개로 분기되는 로직이 필요하나 SPEC에 상세 전환 플로우 없음.
- **유산 후 재임신 시나리오**: '기록 멈춤' 상태의 Pregnancy가 있는 상태에서 새 임신을 등록할 경우, 두 Pregnancy 엔티티 간의 상태 충돌 처리 정책 없음.

### A-3. CloudKit 가족 공유 동시편집 충돌

**위험도: 높음**

- **동시 편집 충돌 해소 전략 미정의**: 부모 두 명이 동시에 아이 성장 기록(키/몸무게)을 입력할 경우, CoreData + CloudKit의 기본 충돌 정책은 "last-write-wins"이나, 이를 명시적으로 수용할지 또는 사용자에게 충돌 해소 UI를 보여줄지 SPEC에 정의 없음.
- **오프라인 편집 후 동기화**: 오프라인 상태에서 A 기기와 B 기기 모두 동일 날짜의 GrowthRecord를 수정하고 나중에 동기화 시, 어느 값이 우선되는지 정책 없음. CloudKit의 CKRecord.lastModifiedUserRecordID 활용이 필요하나 SPEC에 미언급.
- **가족 공유 해제 시 데이터 분리 미정의**: 이혼·별거 시나리오에서 "회수 불가" 정책은 명시되어 있으나(부록 B.5), 해제 후 두 기기의 데이터가 독립적으로 분기될 때 이후 편집분이 상대방 기기에 동기화되지 않도록 막는 로직이 필요. 공유 해제 즉시 CloudKit 구독 제거가 필수이나 구현 주의사항 없음.
- **최대 6인 공유 동시 편집**: 조부모까지 6명이 동시에 접속할 경우 CloudKit 공유 컨테이너의 변경 알림(CKDatabaseSubscription) 폭풍으로 불필요한 네트워크 사용 및 UI 업데이트 과부하 가능.

### A-4. EXIF 날짜 누락 시 Fallback

**위험도: 중간**

- **EXIF 없는 사진의 날짜 선택 UI**: SPEC에 "촬영일 직접 선택 fallback"(2.3)이 명시되어 있으나, 카톡/SNS 수신 이미지처럼 EXIF가 완전히 제거된 경우와 EXIF가 있으나 날짜가 1970-01-01(epoch 0)으로 손상된 경우를 구분해 처리해야 함.
- **EXIF 날짜의 시간대 정보**: 해외에서 촬영된 사진의 EXIF에는 현지 시간이 기록되나 시간대 정보 없이 로컬 시간만 포함된 경우가 많아, 한국 귀국 후 다운로드 시 날짜가 하루 어긋날 수 있음.
- **PHPicker 메타데이터 제한**: iOS 14+의 PHPicker는 프라이버시 보호를 위해 EXIF를 제한적으로 제공함. `PHAsset.creationDate` 접근을 위해 `PHPhotoLibrary` 권한이 필요한데, PHPicker 최소 권한 원칙(SPEC 8.4)과 충돌 가능.
- **타임라인 정렬 기준**: EXIF 날짜, 직접 입력 날짜, CoreData 저장 날짜 세 가지 시점이 혼재될 경우 타임라인 정렬 기준이 명확하지 않으면 같은 날짜의 항목 순서가 기기마다 달라질 수 있음.

### A-5. 시간대·윤년 경계

**위험도: 중간**

- **월령 계산의 월 기준**: "N개월" 계산 시 `Calendar.current.dateComponents([.month], from: birthDate, to: today)`를 사용해야 하나, 2월 생일 아이(예: 1월 31일생)의 경우 2개월 후가 3월 31일인지 3월 28/29일인지에 따라 월령이 달라짐.
- **예방접종 스케줄의 날짜 계산**: "생후 N일에 접종"이 달력 날짜 기준인지 정확히 N×24시간 기준인지 미정의. 특히 서머타임이 없는 한국에서는 문제없으나, 향후 글로벌 확장 시 문제.
- **뱃지 티어 분기 산정**: "최근 6개월 활동 기준, 분기 1회 자동 재산정"(SPEC 7.2)에서 "최근 6개월"의 기산점이 재산정 실행 시점인지 분기 시작일인지 명확하지 않음. 경계일 직전 활동이 포함/제외되는 edge case 발생 가능.

## (b) 민감 영역 엣지케이스

### B-1. '기록 멈춤' 모드의 트리거 조건

**위험도: 최고**

- **트리거 방식 미정의**: SPEC에 "상실 시 기록 멈춤 모드"라고만 되어 있으나, 모드 진입이 사용자가 직접 선택하는 방식인지, 앱이 자동 감지하는 방식인지, 또는 두 가지 모두인지 정의되어 있지 않음. 자동 감지 시 오탐(실제 상실이 아닌데 모드 진입) 위험이 크므로, 자동 감지가 아닌 사용자 선택을 기본으로 해야 할 가능성 높음.
- **모드 진입 후 중단해야 할 항목 목록 미완성**: "주차 알림·태아 가이드·권유 알림" 외에 홈 화면의 임신 주수 카운트다운, 산전 검사 리마인더, 가족 공유 구성원에게 발송되는 알림도 즉시 차단해야 하는지 미정의.
- **가족 공유 구성원에의 알림 차단**: 공유 중인 배우자/조부모 기기에도 동시에 '기록 멈춤' 모드가 적용되어야 하나, CloudKit 동기화 지연(수 분~수십 분) 중 배우자 기기에서 "이번 주 태아가 레몬 크기예요!" 알림이 먼저 도달하는 시나리오가 존재.
- **모드 복귀 조건 미정의**: 기록 멈춤 상태에서 새 임신을 시작할 때 이전 상실 기록은 어떻게 처리하고, Pregnancy 엔티티의 `status` 값(SPEC 1.5)이 어떤 값을 가져야 하는지, `status` enum의 케이스 자체가 SPEC에 정의되어 있지 않음.
- **데이터 보존·삭제 선택권의 UX**: "유저가 기록을 간직하거나 정리할 수 있게"(SPEC 1.4)라고 명시되어 있으나, 삭제 선택 시 CloudKit을 통해 공유 구성원의 기기에서도 즉시 삭제되는지, 아니면 각 기기에서 개별 결정해야 하는지 정책 없음.

### B-2. 미사용 신호 자동 감지의 오탐/누락 위험

**위험도: 높음**

- **"오래 미접속" 기준 미정의**: 미접속 기간이 7일인지 14일인지 30일인지 정의가 없으며, 산전 검사 직후 며칠간 앱을 안 열 수도 있는 정상 패턴과 상실 후 미접속을 어떻게 구분할지 알고리즘 기준 없음.
- **오탐(False Positive) 위험**: 출장·여행·입원으로 인한 일시적 미접속을 상실 신호로 잘못 분류하면, 앱이 조용해진 것에 안도하던 사용자가 복귀 시 '기록 멈춤 모드 해제' 알림을 받아 오히려 충격을 줄 수 있음.
- **누락(False Negative) 위험**: 상실 직후에도 습관적으로 앱을 열어보는 사용자(예: 다른 기능 사용)가 있을 경우, 미접속 신호가 발생하지 않아 주차 알림이 계속 발송될 수 있음.
- **권유 알림 "자동 완화"의 구체적 정의 부재**: "완화"가 빈도를 줄이는 것인지(주 1회→격주), 완전 중단인지, 어떤 카테고리 알림부터 줄이는지 기준 없음.
- **온디바이스 처리 vs 서버 처리**: CLAUDE.md의 "온디바이스 우선" 원칙에 따르면 미사용 신호 감지는 온디바이스로 처리해야 하나, 앱이 완전히 비활성화된 상태(백그라운드 실행 없음)에서는 온디바이스 감지가 불가능하여 서버 기반 "미수신 알림 없음" 신호를 활용해야 함.

### B-3. 이혼·별거 시 가족 공유 해제 후 데이터 처리

**위험도: 높음**

- **"회수 불가" 정책의 구현 복잡성**: 부록 B.5에 "기존 데이터(기기 저장분) 회수 불가 명시"라고 되어 있으나, 이는 CloudKit private + shared database의 특성상 공유 해제 후에도 상대방이 로컬 캐시를 보유하게 됨을 의미함. 이를 UI에서 명확히 사전 고지해야 하는 타이밍과 방식이 미정의.
- **공유 해제 후 이후 신규 기록의 노출 범위**: 해제 직후부터 생성되는 새 GrowthRecord, DiaryEntry 등이 이전 공유 구성원 기기에 동기화되지 않도록 CloudKit 공유 권한을 즉시 변경하는 로직이 필요. 네트워크 오프라인 상태에서 공유 해제 시 권한 변경이 지연되는 gap 구간 대응 필요.
- **자녀 정보 접근권 분쟁**: 이혼 후 양육권 분쟁 상황에서 일방이 앱 내 아이 기록을 삭제하면 상대방 기기에도 삭제 동기화가 될 수 있음. 이는 법적 증거 삭제로 이어질 수 있어 "삭제 불가" 또는 "독립 보존" 정책이 필요하나 SPEC에 없음.
- **Pro 구독자와 공유 해제**: Pro 사용자가 공유 그룹에서 유일한 Pro인 경우, 그룹 해제 후 비-Pro 구성원의 데이터 접근 권한 처리 방식 미정의.

## (c) 예외 UX 상태 점검 대상

### C-1. 빈 상태 (Empty State)

- **아이 프로필 0명**: 온보딩 스킵 후 홈 화면 진입 시 "감정 진입점(아이 프로필 + 최근 사진)"(SPEC 9.1) 영역이 어떻게 표시되는지 명시 없음.
- **기록이 없는 홈**: "주간 성장 리포트" 자동 생성(SPEC 9.4)에서 기록이 0개일 때 fallback 톤은 별도 명시되어 있으나, 생성 자체를 건너뛸지 희소 메시지를 보여줄지 로직 분기가 불명확.
- **크루 밀도 미달 상태**: "기대감 UI"(SPEC 13.5)는 잘 정의되어 있으나, 사용자가 이사 후 밀도 미달 지역으로 이동했을 경우 기존에 참여했던 크루 데이터가 어떻게 표시되는지 미정의.
- **마켓 콜드스타트**: "전국에 얇게 깔지 않고 동네에 집중" 전략인데, 검색 결과가 0개일 때의 empty state가 '없음'인지 '인근 다른 동네 결과'를 보여줄지 기준 없음.

### C-2. 로딩 상태

- **외부 API 로딩 시간**: 건강보험심사평가원 API + 카카오맵 API 두 개를 병렬 호출 시 느린 쪽을 기다리는 동안의 부분 로딩 표시 전략 없음.
- **CloudKit 초기 동기화**: 신규 기기에서 로그인 직후 CloudKit 전체 동기화가 완료되기 전에 앱을 사용할 때 CoreData로컬의 "빈 상태"가 보이면서 동기화 완료 후 갑자기 데이터가 나타나는 "유령 데이터" 현상 대응 필요.
- **WHO 성장 곡선 차트**: 로컬 번들로 제공되므로 로딩 이슈는 없으나, 온디바이스 Core ML 모델 초기 로딩 시 블로킹 여부 확인 필요.

### C-3. 에러 상태

- **외부 API 실패**: 건강보험심사평가원 API 다운 시 소아과 검색이 완전히 불가능해지는 것은 응급 모드의 킬러 기능 가치를 훼손함. 최소한 마지막 성공 캐시 결과를 보여주고 "정보가 N시간 전 기준"임을 명시하는 fallback 필요.
- **카카오맵 API 할당량 초과**: 월 30만 건 무료 한도(SPEC 3.5) 초과 시 키즈카페 검색이 전면 중단됨. 임계값 알림 및 초과 시 대체 표시 로직 필요.
- **복지로 API 오류**: 정부지원금 정보를 못 가져올 경우 "신청 마감 임박" 카드가 홈 2순위 카드로 나오다가 갑자기 사라지는 현상이 발생할 수 있음.
- **질병관리청 API 오류**: 예방접종 스케줄 API 실패 시 로컬 번들 fallback이 있는지 명시 필요.

### C-4. 권한 거부

- **위치 권한 거부 시**: "수동 지역 선택으로 핵심 가치 제공"(SPEC 8.4)이 명시되어 있으나, 수동 선택 UI(읍·면·동 선택 vs 주소 검색)가 SPEC에 정의되어 있지 않음.
- **사진 권한 거부 시**: PHPicker 사용으로 최소 권한이지만, EXIF 날짜 접근을 위해 PHAsset 권한이 추가로 필요한 케이스에서 거부 시의 fallback(수동 날짜 입력) 플로우 확인 필요.
- **알림 권한 거부 시**: 예방접종 D-1 알림이 핵심 기능인데, 거부한 사용자에게 위젯을 대안(SPEC 10.4)으로 제시하는 UI 진입점이 어디인지 미정의. 거부 후 재요청 타이밍 전략 없음.
- **마이크/카메라 권한**: 기능 명세상 카메라 직접 촬영도 포함되는지 명시 없음(PhotosUI만 언급).

### C-5. 오프라인 상태

- **CoreData 로컬 데이터와 CloudKit 동기화 불일치**: 오프라인 중 기록 후 온라인 복귀 시 동기화 충돌을 사용자가 인지할 방법 없음.
- **응급 모드의 오프라인**: 야간 응급 상황에서 오프라인이면 소아과 검색이 불가능. 마지막 캐시 결과를 보여줘야 하나, 캐시 전략 명시 없음.
- **스트리크/연속 기록 뱃지의 오프라인 날짜 처리**: "30일 연속 기록" 뱃지(SPEC 7.3)에서 오프라인으로 기록 후 다음날 동기화되면 동기화 시점 날짜로 기록되어 스트리크가 끊길 위험.

## (e) 디자인/스펙 확정이 필요한 오픈 질문 목록

### OQ-01. [P0] 임신 주수 계산 표준화

- LMP, 초음파 수정 EDD, 직접 입력 EDD 중 주수 계산 우선순위는?
- "주수" 표현이 "12주 3일"인지 "13주차"인지 의학 표준 기준 확정 필요.

### OQ-02. [P0] '기록 멈춤' 모드 트리거 설계

- 사용자가 직접 활성화하는 버튼형인가, 앱이 자동 감지하는가, 또는 둘 다인가?
- 공유 구성원에게 모드가 어떻게 전파되는가? (즉시 동기화 vs 개별 설정)
- 모드 진입 후 차단되는 알림·콘텐츠의 완전한 목록이 필요.

### OQ-03. [P0] Pregnancy → Child 전환의 원자성 보장

- 전환 중 앱 종료/네트워크 단절 시 롤백 또는 재시도 전략은?
- Pregnancy `status` enum의 케이스를 정의해야 함: active / paused(기록 멈춤) / transitioned(전환 완료) / lost(상실) 등.

### OQ-04. [P1] CloudKit 동시 편집 충돌 해소 정책

- "last-write-wins"를 수용할 것인가, 충돌 시 사용자에게 선택권을 줄 것인가?
- 가족 공유 해제 후 오프라인 편집의 동기화 차단 보장 방법은?

### OQ-05. [P1] 외부 API 실패 시 fallback 전략

- 건강보험심사평가원 API 다운 시: 캐시 유효 시간, 캐시 표시 방법, 전화 연결 전면 전환 트리거 조건.
- 질병관리청 API 다운 시: 로컬 번들 스케줄 데이터 포함 여부 및 갱신 주기.
- 카카오맵 API 할당량 초과 시: 대체 지도 API 또는 기능 중단 고지 방식.

### OQ-06. [P1] 미사용 신호 자동 감지 알고리즘

- "오래 미접속"의 정의: 임신 중 미접속 N일 기준.
- 온디바이스 vs 서버 처리 결정: 앱이 완전 비활성화 시 감지 가능한가?
- 감지 후 "완화"의 구체적 정의: 어떤 알림을 어느 빈도로 줄이는가?

### OQ-07. [P1] 이혼·별거 후 데이터 처리 UI

- 공유 해제 사전 고지 시점과 메시지 문구.
- 아이 기록 삭제 시 공유 구성원 기기에서의 동기화 삭제 여부 (데이터 주권 vs 증거 보전 충돌).
- Pro 유일 구독자가 그룹 해제 시 비-Pro 구성원의 데이터 다운그레이드 처리.

### OQ-08. [P1] 오프라인 기록의 날짜 기준

- 오프라인 상태에서 기록한 시점 날짜(기기 로컬 시각)를 서버 동기화 시점이 아닌 기록 시점으로 보존하는 방법.
- 연속 기록 뱃지 판정 시 동기화 시점이 아닌 기록 시점 날짜 사용 여부.

### OQ-09. [P2] 다태아(쌍둥이) 전환 플로우

- Pregnancy 1개 → Child N개 전환 시의 데이터 분배 방법.
- 쌍둥이 EDD 계산 기준(단태아 280일 동일 적용 vs 37주 기준 단축).

### OQ-10. [P2] EXIF 날짜 접근 권한 전략

- PHPicker(최소 권한)와 PHAsset.creationDate(전체 라이브러리 권한) 간의 trade-off 결정.
- 손상된 EXIF(1970-01-01, 미래 날짜) 처리 기준 정의.

### OQ-11. [P2] 성장 카드 Dynamic Type 대응

- ImageRenderer로 생성한 정적 이미지에 Dynamic Type 적용 방법.
- 카드 생성 시 접근성(VoiceOver용 alt text) 처리 방법.

### OQ-12. [P2] 게스트 → 회원 마이그레이션

- 게스트 상태에서 로컬 저장된 CoreData 데이터를 회원 가입 시 계정에 연결하는 마이그레이션 전략.
- 마이그레이션 실패 시 데이터 손실 방지 롤백 방법.
