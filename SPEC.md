# 기능 1. 임신 기록 & 태아 일지

> **[정식 기능]** 앱의 시작점을 '출산 후'에서 '임신 확인 순간'으로 당기는 핵심 획득 기능. 출산 전 락인 → 성장 기록으로 자연 전환.

## 1.1 기능 목적

임신은 부모가 가장 설레고 불안한 시기다. 태아의 성장, 산모의 몸 변화, 산전 검사 일정을 기록하고 싶지만 지금은 다이어리·캘린더·맘카페에 흩어져 있다. 임신 기록을 정식 기능으로 제공하면 출산 9~10개월 전부터 유저를 확보하고, 출산 시점에 자연스럽게 아이 성장 기록으로 이어진다. 획득과 리텐션을 동시에 강화하는 입구.

## 1.2 핵심 기능 스펙

| 항목 | 스펙 |
| --- | --- |
| 필수 입력 | 마지막 생리일(LMP) 또는 출산 예정일(EDD) |
| 자동 계산 | 임신 주수(D+주차), 출산까지 D-day, 현재 삼분기 |
| 선택 입력 | 태명, 산부인과, 다태아 여부(쌍둥이 등) |
| 다태아 지원 | 쌍둥이·삼둥이 각각 또는 통합 기록 |
| 가족 공유 | 배우자·예비 조부모와 공유(출산 전부터 가족 유입) |

| 데이터 소스 | 공개 의학 정보 기반 자체 큐레이션(의료 조언 아님) |

| 면책 고지 | 일반 정보이며 의료 상담을 대체하지 않음 명시 |

| 배 사진(D라인) | 주차별 배 사진 타임라인 — 출산 후 성장 사진과 연속 |

## 1.3 임신 → 출산 전환 (핵심 연결)

임신 기록의 가장 큰 가치는 출산 시점에 아이 성장 기록으로 매끄럽게 이어진다는 점이다. 출산 예정일 전후로 '아기가 태어났나요?' 전환 플로우를 제공해, 임신 프로필을 아이 프로필로 승계한다.

- 배 사진 타임라인 → 아이 성장 사진 타임라인으로 연속(끊김 없는 여정)

- 태명 → 아이 이름으로 전환, 예정일 → 실제 생년월일 확정

- 임신 기록은 그대로 보존 — '태아 시절' 아카이브로 평생 남김

- 출산 후 예방접종·성장 기록(기능 2)으로 자연 진입

## 1.4 엣지 케이스 배려 (민감)

임신은 유산·사산 등 상실의 가능성이 있는 가장 민감한 영역이다. 무신경한 알림 하나가 깊은 상처가 될 수 있으므로 최고 수준의 배려가 필요하다. 부록 B 엣지 케이스 정책과 연동.

- '기록 멈춤' 모드 — 상실 시 모든 주차 알림·태아 가이드 즉시 중단

- 미사용 신호 자동 감지 — 오래 미접속 시 권유 알림 자동 완화(직접 설정 부담 제거)

- 따뜻하고 중립적인 카피 — 상실 가능성을 염두에 둔 표현, 닦달 절대 금지

- 데이터 보존·삭제 선택권 — 유저가 기록을 간직하거나 정리할 수 있게

## 1.5 데이터 구조

| --- | --- |

## 1.6 기술 스택

- 로컬 저장: CoreData + CloudKit(가족 공유, 출산 전부터)

- 주차 계산: LMP/EDD 기반 임신 주수 산출 로직

- 주차별 콘텐츠: 자체 큐레이션 데이터(로컬 번들)

- 차트: Swift Charts(체중 추이)

- 이미지: PhotosUI + async 압축(배 사진 타임라인)

## 1.7 성공 지표

# 기능 2. 성장 기록 & 육아 일지

> **[정식 기능]** 앱의 핵심 리텐션 기능. 아이 데이터가 축적되면 이탈율 극히 낮음. 초경량 입력(사진=1기록)으로 재설계.

## 2.1 기능 목적

현재 사용자들은 아이 성장 기록을 인스타그램, 카톡 앨범, 메모장에 분산 저장하고 있음. 하나의 타임라인으로 통합하여 아이의 성장 여정을 완전히 보존하는 것이 목적. 임신 기록(기능 1)에서 출산 시점에 자연스럽게 이어져, 태아 시절부터 성장까지 끊김 없는 하나의 여정을 형성한다.

## 2.2 핵심 기능 스펙

### 2.2.1 아이 프로필

| 항목 | 스펙 |
| --- | --- |
| 프로필 수 | 최대 5명 (다자녀 지원) |
| 필수 입력 | 이름, 생년월일, 성별 |
| 선택 입력 | 프로필 사진, 태명, 출생 병원 |
| 자동 계산 | 월령 자동 표시 (D+N일, N개월, N살) |
| 가족 공유 | CloudKit 기반, 최대 6명 공유 (아빠 포함) |

### 2.2.2 성장 측정 기록

| 차트 | WHO 성장 곡선 오버레이, 또래 백분위 표시 |

## 2.3 기록 경험 설계 (토론 반영)

워킹맘은 매일 측정할 시간이 없고, 기록이 숙제처럼 느껴지면 이탈한다. 반면 기록 덕후는 풍부한 데이터를 원한다. 두 니즈를 온보딩 프리셋으로 분기한다. R1·R22 반영.

### 초경량 입력 (기본)

- 사진 1장 = 1기록 — FAB 탭 → 사진 선택 → 저장, 2탭으로 완료

- 키/몸무게는 선택 입력, 사진 EXIF 날짜로 타임라인 자동 정렬

- EXIF 없는 사진(카톡 수신 등)은 '촬영일 직접 선택' fallback

- 저장 후 작은 보상 — 따뜻한 애니메이션, '○○이의 152번째 순간'

- 과거 날짜 소급 입력 — 달력에서 톡으로 간편 추가(몰아서 기록 대응)

### 자세히 모드 (펼치기)

- 빠른 기록 후에도 추가 편집 가능, 강제 단계 0

### 고급 기록 모드 (옵션)

- 수유량·수면 시간·배변 트래킹(신생아 데이터, 소수 니즈)

- recordType enum + customFields JSON으로 스키마 확장 대비

### 다자녀 배려

- 상단 아바타 탭으로 아이 빠른 전환

- 둘째 기록이 뜸하면 '둘째 ○○이 기록이 뜸해요' 부드러운 리마인드(둘째 서러움 케어)

### AI 일지 초안 (Pro)

- 사진 기반 캡션 자동 작성으로 글쓰기 부담 제거(기록률 직결)

- 초안 제시 → 한 번에 편집, AI 티 안 나는 자연스러운 톤

## 2.4 성장 카드 공유 (인스타용 바이럴)

TickLab의 측정 결과 공유 카드를 육아 버전으로 적용. 아이 사진 위에 최근 키·몸무게 등 성장 데이터를 우아하게 오버레이해 인스타에 바로 올릴 수 있는 카드를 생성한다. 엄마들이 자랑하고 싶어하는 콘텐츠라 자연 바이럴이 강력하고, 워터마크가 곧 광고가 된다. R7 공유 바이럴 논의와 연결.

### 2.4.1 카드 구성

| 요소 | 스펙 |
| --- | --- |
| 배경 | 선택한 아이 사진(전체 화면) |
| 스크림 | 하단 그라데이션 오버레이 — 어떤 사진에도 텍스트 가독성 확보 |
| 데이터 위치 | 기본 왼쪽 하단, 위치 변경 가능(아이 얼굴 가림 방지) |
| 브랜드 마크 | 우측 상단 BabyLog 로고(워터마크) |
| 비율 | 피드 4:5, 정사각 1:1, 스토리 9:16 지원 |

### 2.4.2 데이터 배치 위치

사진마다 아이 얼굴이나 중심 피사체 위치가 다르므로 데이터 오버레이 위치를 선택할 수 있다.

- 왼쪽 하단(기본) / 오른쪽 하단 / 왼쪽 상단 / 중앙 하단 / 표시 안 함

## 3.4 응급 모드 (킬러 기능)

아이가 아플 때 야간/주말 진료 소아과를 찾는 것이 가장 큰 페인포인트이자 첫 다운로드 동기. 정보가 틀리면 한 번에 신뢰를 잃으므로 신뢰성 설계가 핵심. R2 반영.

### 응급 모드 동작

- 홈의 응급 모드 버튼 → 탭 한번에 현재 영업중 + 거리순 소아과 즉시 표시

- 필터 조작 없이 '지금 갈 수 있는 곳'이 바로 노출

- 전용 고대비 다크 레이아웃 — 한밤중·불안한 부모·어두운 환경 최적화

- 리스트 퍼스트(지도는 로딩 느림), 큰 터치 타깃, 최소 정보

- 전화 버튼 최대 크기 — 초기엔 앱이 판단 않고 '전화로 확인' 유도(데이터 공백 우회)

### 신뢰성 장치

| 장치 | 내용 |
| --- | --- |
| 마지막 확인 시각 | 영업정보에 '○분 전 확인' 표시 + 신뢰도 등급 |
| 유저 크라우드소싱 | '지금 문 닫혔어요' 신고로 실시간 보정 |
| 초기 콜드스타트 | 유저 부족 시 전화 연결 전면 배치 → 데이터 쌓이며 점진 자동화 |
| 면책 고지 | 영업 정보 신뢰도 한계 명시 |

### 응급 → 기록 퍼널

응급 모드만 쓰고 나가는 유저를 잡기 위해, 응급 상황 종료 후 '○○이 오늘 진료 기록 남길까요?'로 기록 기능에 연결한다. 응급 → 진료 기록 → 성장 일지로 이어지는 퍼널.

### 즐겨찾기 병원 위젯

단골 소아과가 정해진 유저를 위해, 즐겨찾기 병원 3곳의 오늘 진료 여부를 홈 화면 위젯으로 제공. 매번 검색할 필요 없음.

# 기능 4. 육아 중고 마켓

> **[정식 기능]** 바이럴 핵심 기능. 성장 졸업 → 매물 공급 엔진으로 차별화. 수수료 수익원.

## 4.1 기능 목적

육아용품은 사용 기간이 극히 짧음(신생아 옷 3개월, 바운서 6개월). 당근마켓은 범용이라 월령 필터, 안전 정보, 엄마들간 신뢰 거래에 최적화되어 있지 않음. 육아용품 특화 C2C 마켓 공백 존재.

## 4.2 차별화 포인트

| 기능 | 당근마켓 대비 차별점 |
| --- | --- |
| 월령 태그 | 0-3개월, 3-6개월 등 필수 태그로 정확한 매칭 |
| 카시트 리콜 DB | 리콜 제품 자동 경고 표시 |
| 상태 등급 표준화 | S/A/B/C 등급 체크리스트 기반 |
| 엄마 인증 | 아이 프로필 연동 = 실제 육아 사용자 신뢰도 |
| 렌탈 기능 | 구매 대신 단기 렌탈 (바운서·스윙 등 단기 사용품) |

## 4.3 핵심 전략 — 매물 공급 엔진 (토론 반영)

당근 대비 매물 수 열세는 치명적이다. 마켓을 독립 경쟁시키면 진다. 핵심은 '기록 앱 유저가 졸업한 물건을 자연스럽게 내놓는' 흐름. 마켓을 별도 앱이 아니라 기록 앱의 자연스러운 부산물로 설계한다. R3 반영.

- 성장 졸업 트리거 — 성장 기록에서 '이 시기 끝났어요 → 쓰던 물건 정리할까요?'로 공급 생성

- 동네 졸업템 컨셉 — 같은 단지/동네 양육자가 쓰던 것이라는 신뢰

- 월령 연동 '곧 필요해요' 피드 — 아이 월령 기준 곧 필요할 물건 추천(8개월→보행기·이유식 의자)

- 위생 상태 셀프 체크리스트 사진 필수(젖병·카시트류)

### AI 자동 등록 (마찰 제거)

팔 시간이 없어 당근에 안 올리는 다수를 위해, 사진을 찍으면 품목을 자동 인식해 등록 부담을 줄인다. 단 비용·정확도 고려해 단계적으로.

| 단계 | 범위 |
| --- | --- |
| 1단계 | 온디바이스 비전 카테고리 자동 분류(비용 0) |
| 2단계 | 비슷한 매물 평균 기반 시세 제안(거래 데이터 축적 후) |
| 주의 | 브랜드/모델 식별은 정확도 한계 → 사용자 확인 |

연결은 은근하게(있으면 보이는 정도), 푸시·팝업으로 들이밀지 않는다. 정보가 주인공, 매물은 조연(콘텐츠 커머스). 광고처럼 느껴지는 순간 신뢰가 무너진다.

# 기능 5. 육아 가계부

## 5.4 입력 부담 제거 설계 (토론 반영)

가계부 앱을 따로 쓰는 사람에게 또 입력하라면 안 쓴다. 이중 입력은 절대 거부 대상. 가계부를 '입력하는 곳'이 아니라 '보여주는 곳'으로 만든다. R5 반영.

- 앱 내 거래 자동기록 — 마켓 거래·구독·인쇄 결제가 자동으로 가계부에 흘러들어옴

- 유저는 가끔 큰 지출만 수동 추가 — 입력 부담 0에 가깝게

- 자동 수집 대시보드 중심 — 수동 입력 최소화

### 부부 공유

- 가계부도 부부 공유 — 육아비 누가 얼마 썼는지 투명하게

- 지출자 태그로 구분

### 정부지원금 — 알림을 넘어 신청까지

지원금 알림은 강력한 획득 훅이자 공유가 잘 되는 기능. '알림'만 주고 끝이 아니라 신청까지 안내한다.

# 기능 6. 동네 육아 크루

> **[정식 기능]** 네트워크 효과 극대화. 동네 밀도 게이팅으로 콜드스타트 해결.

## 6.1 기능 목적

비슷한 월령의 아이를 키우는 동네 부모 연결. 같이 키즈카페 가기, 공동육아, 정보 공유. 맘카페 커뮤니티의 앱 버전. 당근마켓이 중고거래→동네생활로 확장한 것처럼 BabyLog의 핵심 해자가 될 수 있음.

## 6.2 매칭 로직

| 매칭 기준 | 상세 |
| --- | --- |
| 위치 | 반경 2km 이내 (사용자 설정 가능, 1~5km) |
| 월령 유사도 | 아이 나이 ±3개월 이내 |
| 활동 성향 | 실내선호/야외선호/무관 선택 |
| 가입 조건 | 아이 프로필 등록 필수 (신뢰도 확보) |
| 익명성 | 닉네임 + 아이 월령만 공개, 실명 비공개 |

## 6.3 핵심 기능

### 6.3.1 크루 찾기

- 지도뷰 — 동네 부모 프로필 핀 표시 (위치 대략적으로 표시, 정확 주소 비공개)

- 리스트뷰 — 유사 월령순 정렬

- 관심사 태그 — '산책 즐겨요', '책 많이 읽어요', '놀이터 자주 가요' 등

| --- | --- |

## 6.4 기능 연결 & 커뮤니티 운영 (토론 반영)

크루에서 만난 양육자와 중고거래도 자연스럽게 이어진다. 커뮤니티는 운영을 못 하면 독이 되므로(맘카페 분란·광고 도배·정치 논쟁) 단계적으로 키운다. R12·R18 반영.

### 크루 ↔ 마켓 연동

- 같은 크루 멤버 매물 우선 노출(안심 거래)

- 연락처 교환 없이 앱 내에서 거래·만남 연결

### 경량 → 풀 커뮤니티 2단계

게시판형 풀 커뮤니티는 운영 리스크가 가장 크다. 그 전에 경량 커뮤니티(댓글·동네 정보 공유)로 운영 역량을 먼저 검증한 뒤 풀 커뮤니티를 연다.

| 단계 | 범위 |
| --- | --- |
| 경량 | 댓글·동네 정보 공유로 운영 역량 선검증 |
| 풀 | 게시판형 커뮤니티(운영 역량 확보 후) |

### 모더레이션 (솔로 운영)

- 명확한 커뮤니티 규칙 + 민감 주제(정치·종교·논쟁) 차단

- 자동 금칙어 필터 + 신고 임계값 자동 블라인드 + AI 사전 필터링

- 시드 유저 큐레이션 — 첫 100명이 따뜻하면 분위기가 따뜻하게 유지

- 공감 유도 UI, 분란 유발 기능(비추천·논쟁 정렬) 배제

- 양질 정보 큐레이션 — 눈팅족도 만족하는 읽을거리

## 6.5 안전 & 신뢰 정책

- 신규 가입 시 전화번호 인증 필수

# 기능 7. 뱃지 & 신뢰도 시스템

  TickLab 뱃지 시스템을 육아 맥락으로 재설계. 당근 온도처럼 숫자 하나가 아니라 활동 성격을 언어로 표현.

## 7.1 시스템 목적

거래 신뢰도 확보 + 커뮤니티 기여 인정 + 유저 락인 강화. 뱃지는 '이 사람이 어떤 유저인지'를 한눈에 보여주는 언어. 골든 맘 · 나눔 천사 · 육아고수 조합이 그 사람의 플랫폼 내 정체성이 됨.

## 7.2 티어 구조

티어는 최근 6개월 활동 기준으로 분기 1회 자동 재산정.

### 새싹 티어

| 항목 | 내용 |
| --- | --- |
| 조건 | 가입 직후 ~ 거래 2회 이하 |
| 대표 뱃지 | 새싹 엄마 (기본) / 첫 거래 완료 (첫 거래 후 자동 부여) |
| 색상 | 그레이 계열 (#F1EFE8) |
| 목적 | 신규 유저 환영, 첫 활동 독려 |

### 따뜻한 이웃 티어

| 항목 | 내용 |
| --- | --- |
| 조건 | 거래 3~9회 + 후기 보유 |
| 대표 뱃지 | 따뜻한 이웃 |
| 획득 가능 뱃지 | 나눔 천사 (무료나눔 3회+) / 빠른 답장 (응답률 90%+) |
| 색상 | 민트/그린 계열 (#E1F5EE) |
| 목적 | 활발한 거래 유저 식별 |

### 믿음직한 맘 티어

| 항목 | 내용 |

| 획득 가능 뱃지 | 육아고수 (기록 50회+) / 정보통 (게시글+후기 20건+) / 안심 판매자 |

| 1주년 | 가입 1주년 달성 | 매년 갱신 |
| Pro 멤버 | Pro 구독 활성 상태 | 구독 유지 중 표시 |

## 7.4 프로필 카드 구성

프로필 카드는 아이디 옆에 티어 뱃지를 메인으로 표시하고, 하단에 보조 뱃지를 최대 3개까지 노출한다.

### 프로필 카드 레이아웃

| 영역 | 내용 |
| --- | --- |
| 아바타 | 프로필 사진 또는 닉네임 이니셜 원형 (48px) |
| 닉네임 옆 | 티어 뱃지 1개 필수 표시 (골든 맘 / 믿음직한 맘 등) |
| 프로필 부제 | 아이 월령 · 지역(동 단위) · 가입 기간 |
| 뱃지 줄 | 보조 뱃지 최대 3개 (유저가 순서 직접 설정) |
| 통계 줄 | 거래 횟수 · 후기 평점 · 크루 모임 수 · 응답률 |

### 프로필 카드 예시

| 김지수맘   [골든 맘] 36개월 아이 · 서울 마포구 · 가입 8개월 [안심 거래왕]  [나눔 천사]  [육아고수]  [초기 멤버] 거래 47회  │  후기 평점 4.9  │  크루 12회  │  응답률 94% |
| --- |

## 7.5 매물 리스트 미니 노출

매물 리스트에서는 공간 절약을 위해 뱃지를 최대 2개만 표시한다. 티어 뱃지(1) + 가장 신뢰도 높은 활동 뱃지(1) 조합.

### 미니 노출 예시

| 스토케 트레일즈 · S등급 · 12개월+    45,000원 김지수맘   [골든 맘]  [나눔 천사] |
| --- |
| 맥클라렌 유모차 · A등급 · 0-12개월    68,000원 박하늘맘   [따뜻한 이웃]  [빠른 답장] |

*노출 우선순위 로직: 티어 뱃지 → 거래 신뢰 뱃지 → 응답률 뱃지 순서로 자동 선택*

## 7.6 뱃지 시스템 운영 규칙

### 노출 규칙

| 화면 | 노출 뱃지 수 |
| --- | --- |
| 프로필 상세 | 티어 1 + 보조 최대 3개 (유저 직접 선택) |
| 매물 리스트 | 최대 2개 (티어 + 신뢰 뱃지 자동 선택) |
| 채팅 화면 | 닉네임 옆 티어 뱃지 1개만 |
| 크루 매칭 카드 | 티어 1 + 활동 1개 |

### 갱신 규칙

- 티어 뱃지: 분기 1회 자동 재산정 (최근 6개월 활동 기준)

- 활동 뱃지: 조건 달성 즉시 자동 부여

- 특별 뱃지(베타 테스터, 초기 멤버 등): 영구 보존, 박탈 불가

- Pro 멤버 뱃지: 구독 활성 상태에서만 노출, 해지 시 자동 비활성화

### 강등 및 정지 규칙

- 거래 분쟁 1회 발생 시: 해당 분기 티어 산정에서 거래 횟수 -3회 감산

- 신고 누적 3회 이상: 티어 1단계 강등 + 뱃지 30일 일시 정지

- 허위 정보(리콜 은폐, 상태 허위 기재) 확인 시: 모든 뱃지 영구 박탈

- 플랫폼 신뢰도 유지가 뱃지 시스템의 최우선 목적

### UX 규칙

- 새 뱃지 획득 시: 푸시 알림 + 앱 내 축하 애니메이션 (TickLab 뱃지 언락 경험 이식)

- 프로필 편집: 보조 뱃지 표시 순서 유저가 직접 드래그 설정

- 뱃지 탭: 내가 보유한 뱃지 전체 + 미획득 뱃지 잠금 상태로 표시 (수집 욕구 자극)

- 티어 승급 시: 전용 축하 화면 + 소셜 공유 카드 생성

# 기능 8. 온보딩 & 첫 사용자 경험

페르소나별로 기대가 다르므로(빠름 vs 꼼꼼), 첫인상은 단일 통합 메시지로 시작해 온보딩에서 개인 맞춤으로 분화한다. R6·R26·R27 토론 반영.

## 8.1 설계 원칙

- 프로그레시브 프로파일링 — 필수 입력은 1개(아이 생일 또는 위치)로 제한, 나머지는 사용 중 점진 수집

- 모든 온보딩 단계 스킵 가능 — 강제 입력 0

- 게스트 둘러보기 우선 — 가입 없이 핵심 가치 체험, 로컬 보관 후 가입 시 데이터 마이그레이션

- Aha moment 이원화 — 주변 인프라(즉시 효용=첫 훅) + 개인화 타임라인(첫 리텐션)을 세션1에 모두 경험

## 8.2 온보딩 플로우

| 단계 | 내용 |
| --- | --- |
| 스플래시 | 통합 한 줄 메시지 '우리 동네 육아의 모든 것' |
| 가치 미리보기 | 가입 없이 동네 야간 소아과 즉시 표시(게스트) |
| 기록 밀도 선택 | '가볍게 / 꼼꼼히' 1문항으로 기본 UI 프리셋 결정 |
| 아이 등록 | 생일 입력 → D+일·개월수·예방접종 타임라인 자동 생성 |
| 다자녀 확인 | '아이가 더 있나요?' 한 번에 다자녀 등록 |
| 권한 요청 | 프리퍼미션 다이얼로그로 맥락 설명 후 위치/알림 요청 |

## 8.3 첫 기록 유도

첫 기록 경험('이렇게 쉽구나')이 습관 형성의 시작점이므로, 온보딩 말미에 첫 기록을 자연스럽게 유도한다. 사진 1장이면 완료되는 초경량 경험으로 설계.

## 8.4 권한 요청 전략

| 권한 | 요청 시점 / 맥락 |
| --- | --- |
| 위치 | 주변 가치 미리보기 직전, '근처 소아과를 보여드릴게요' |
| 알림 | 가치 경험 후, '접종일을 놓치지 않게 알림 받을까요?' |
| 사진 | 첫 기록 시도 시 PHPicker로 최소 권한 |

게스트 데이터는 로컬 보관 후 가입 시점에 계정으로 마이그레이션한다. 위치 권한 거부 시에도 수동 지역 선택으로 핵심 가치 제공.

# 기능 9. 홈 화면 & 리텐션 루프

홈은 매 세션 첫 화면으로 리텐션·전환의 전쟁터다. '대시보드'가 아니라 '오늘의 한 장면'으로 설계한다. R7·R21 토론 반영.

## 9.1 홈 화면 구성

| 영역 | 내용 |
| --- | --- |
| 감정 진입점 | 아이 프로필 + 최근 사진 최상단 고정(매일 여는 정서적 이유) |
| 다자녀 전환 | 상단 아이 전환 칩 또는 둘 다 표시 |
| 단일 우선 카드 | 우선순위 엔진이 선정한 '지금 가장 중요한 것' 하나만 크게 |
| 하위 모듈 | 정보/감정 카드 믹스, 모듈 3~4개로 제한, 카드당 단일 액션 |

## 9.2 우선순위 엔진

최상단 단일 카드를 선정하는 룰 베이스 엔진으로 시작해 점차 학습형으로 발전. 홈 모듈은 원격 구성(서버 설정)으로 앱 업데이트 없이 우선순위 조정 가능.

| 우선순위 | 카드 예시 |
| --- | --- |
| 1순위 | 예방접종 임박(D-1, 당일) |
| 2순위 | 정부지원금 신청 마감 임박 |
| 3순위 | 응급 상황 감지(야간 + 최근 검색) |
| 4순위 | '1년 전 오늘' 추억 카드(정서 리텐션) |
| 5순위 | 동네 새 매물 / 크루 활동 |
| 6순위 | 기록 권유(부드럽게) |

## 9.3 리텐션 루프 설계

리텐션을 기록에만 의존하지 않고 비기록 훅을 함께 확보한다. 기록 안 하는 다수 유저도 매일 열 이유를 최소 2개 제공.

- 발견의 재미 — 매일 다른 또래 정보 카드('오늘 ○○이 또래는 이 시기에 이런 걸 한대요')

- 동네 피드 — 새 매물·크루 활동 등 동네 소식

- 주간 성장 리포트 — 키 변화·사진 모음·이정표 자동 생성, 데이터 희소 유저용 fallback 톤 별도

- '1년 전 오늘' 추억 — 정서적 리텐션의 강력한 훅

- 죄책감 대신 발견 — 기록 안 한 날을 비난하지 않고 따뜻하게 유도

## 9.4 주간 리포트 & 공유 바이럴

주간 리포트는 인스타에 올리고 싶을 만큼 예쁜 비주얼로 제작해 자연 바이럴을 유도한다. 아이 사진이 포함되므로 공유 시 프라이버시 옵션(얼굴 가리기, 정보 선택)을 내장한다.

| 항목 | 스펙 |
| --- | --- |
| 생성 주기 | 주 1회 자동 |
| 구성 | 이번 주 사진 베스트, 키/몸무게 변화, 달성 이정표 |
| 희소 fallback | 기록 적어도 '사진 1장도 소중한 순간' 긍정 톤 |
| 공유 | 워터마크 포함 카드, 프라이버시 컨트롤 내장 |

# 기능 10. 알림 전략

알림은 리텐션 도구이자 이탈 1순위 원인이다. 첫 알림 경험이 '유용했다'여야 하므로 초기에는 고가치 알림만 발송한다. R7·R16 토론 반영.

## 10.1 알림 원칙

- 중요도 2단계 분리 — 필수(접종/지원금)는 항상, 권유(기록/콘텐츠)는 끌 수 있게

- 초기 고가치 우선 — 첫인상 보호를 위해 접종·지원금 알림만, 권유 알림은 애착 형성 후

- 알림 권한은 가치 경험 후 맥락과 함께 요청(가입 즉시 요청 금지)

- 알림 카피는 따뜻하게 — 기계적 표현('기록 알림') 금지, '○○이의 오늘이 궁금해요' 식

## 10.2 활동 시간 학습

유저가 앱을 여는 시간대를 온디바이스로 학습해(프라이버시 보전) 열어볼 가능성이 높은 시간에만 발송. 데이터가 쌓이기 전 초기에는 저녁 8~10시 황금시간대 디폴트.

## 10.3 가족 알림 라우팅

부부가 함께 쓸 때 동일 알림(접종일 등)이 양쪽에 중복 발송되지 않도록 역할 분담. 누가 어떤 알림을 받을지 가족 설정에서 지정.

## 10.4 위젯 (알림 백업 채널)

알림을 거부한 유저를 위한 수동 정보 채널. 홈 화면 위젯으로 오늘 할 일(접종·지원금 마감)을 푸시 없이 제공. WidgetKit 구현, 배터리 고려한 업데이트 주기 최적화.

| 위젯 종류 | 표시 내용 |
| --- | --- |
| 오늘의 할 일 | 접종 예정·지원금 마감 |
| 아이 요약 | 최근 사진 + 월령 + D+일 |
| 주변 응급 | 현재 영업 중 소아과 유무 |

# 기능 11. AI 활용

AI는 '기능'이 아니라 비용 구조에 따른 전략으로 접근한다. 온디바이스로 가능한 것은 무료, 서버 LLM이 필요한 것은 Pro로 분기. 'AI' 단어를 전면에 내세우기보다 '마법처럼 알아서 되는' 매끄러운 경험으로 내재화. R17 토론 반영.

## 11.1 비용 분기 전략

| 구분 | 기능 |
| --- | --- |
| 온디바이스 (Core ML) | 사진 자동 분류, 베스트 컷 선별, 월간 자동 앨범 |
| 서버 LLM | 사진 기반 일지 캡션 자동 생성 |
| 서버 LLM | 육아 정보 큐레이션 |

## 11.2 AI 사진 큐레이션 (무료)

- 비슷한 사진 자동 그룹핑

- 잘 나온 베스트 컷 추천

- 매달 자동 앨범 생성

온디바이스 비전으로 처리해 비용 0, 사진이 외부로 나가지 않아 프라이버시 보전.

## 11.3 AI 일지 초안 (Pro 전환 훅)

글쓰기를 부담스러워하는 다수 유저의 기록 장벽을 제거하는 킬러 기능. 사진을 보고 캡션 초안을 생성해 기록률을 직접적으로 끌어올린다. Pro 전환의 강력한 훅.

- 사진 기반 캡션 초안 자동 작성

- 초안 제시 → 한 번에 편집(수정 쉽게)

- AI 티 안 나는 자연스러운 톤

## 11.4 의료 Q & A — 보류 및 대체

'37.5도인데 병원 가야 하나' 같은 의료 질의는 잘못된 조언 시 법적 리스크(소송 가능)가 커서 직접 답변은 보류한다. 대신 검증된 출처(공신력 있는 의료 정보)를 연결하는 큐레이션으로 안전하게 대체.

| 접근 | 처리 |
| --- | --- |
| 직접 의료 조언 | 제공하지 않음(법적 리스크) |
| 증상 관련 질의 | '병원 방문을 권장합니다' + 주변 소아과 연결 |
| 일반 육아 정보 | 검증된 출처 큐레이션으로 제공 |

# 기능 12. 접근성 & 사용 환경

육아는 한 손으로 아이를 안고, 새벽에, 다양한 세대가 함께 쓰는 특수 환경이다. 접근성은 시장 확장과 직결된다. R13 토론 반영.

## 12.1 한 손 조작 & 야간 환경

- 완전 한손 조작 — 중요 버튼을 모두 엄지가 닿는 하단 중심에 배치

- 야간 초저휘도 모드 — 새벽 수유 시 아이가 깨지 않도록 별도 오버레이 레이어

- FAB는 iOS HIG 준수 위치(탭바와 겹침 방지)

## 12.2 색약 & 시각 접근성

- 색 + 아이콘 + 레이블 3중 인코딩 — 색만으로 정보 전달 금지

- 티어 뱃지는 4색으로 제한, 활동 뱃지는 아이콘으로 구분(인지 부하 감소)

- Dynamic Type 연동 — 시스템 글씨 크기 반영

- VoiceOver 완전 대응

## 12.3 조부모 심플 뷰

조부모 세대도 손주 사진을 볼 수 있어야 가족 공유가 의미를 갖는다. 조부모는 곧 신규 유저 유입 채널이자 가족 확장 바이럴 채널. 별도 앱이 아닌 앱 내 '심플 뷰' 토글로 사진 타임라인만 큼직하게 제공.

| 요소 | 심플 뷰 처리 |
| --- | --- |
| 글씨 | 큰 글씨 모드 |
| 화면 | 사진 타임라인 중심, 복잡한 기능 숨김 |
| 조작 | 탭 위주, 최소 인터랙션 |

## 12.4 구현 방침

Dynamic Type·VoiceOver·색약 대응은 iOS 기본기로, 초기부터 시맨틱 컬러 + SF Symbols + Dynamic Type로 설계하면 추가 공수가 거의 없다. 나중에 소급 적용하면 전면 재작업이 되므로 처음부터 내재화한다.

# 기능 13. 신뢰·안전 & 콜드스타트

아이 관련 서비스에서 안전은 타협 불가 영역이다. 사고 한 건이 브랜드를 죽일 수 있다. 동시에 신뢰는 높이되 거래 마찰은 최소화하는 균형이 핵심. R8·R14 토론 반영.

## 13.1 거래 안전

- 거래 후기 의무화 + 노쇼 페널티

- 공공장소 안심 거래존 — 주민센터·소아과 앞 등 안전한 거래 장소 추천(공공데이터)

- 본인인증 — 첫 거래 시 1회만(PASS/휴대폰 인증), 마찰 최소화

- 신고 버튼 상시 노출하되 비침습적

## 13.2 크루 만남 안전

- 크루 본인인증 의무 + 상호 평가 + 신고 시스템

- 아이 프로필 없는 계정은 크루 기능 접근 불가

- 성별 필터 — 여성 회원 전용 크루 옵션

- 위치 정보는 동/구 단위만 공개, 정확한 주소 비공개

## 13.3 분쟁 처리 (솔로 운영 대응)

솔로 개발자가 모든 CS를 감당할 수 없으므로 운영 부담을 낮추는 자동화가 생존 조건. 신고 누적 자동 제재 + 커뮤니티 자정 시스템으로 초기 운영.

| 상황 | 자동 처리 |
| --- | --- |
| 신고 누적 3회 | 자동 블라인드 + 티어 1단계 강등 + 뱃지 30일 정지 |
| 허위 정보(리콜 은폐 등) | 모든 뱃지 영구 박탈 |
| 거래 분쟁 1회 | 분기 티어 산정 시 거래 -3회 감산 |

안전을 강조하되 '위험한 앱인가?' 인상을 주지 않도록 공포 조장 톤은 회피한다.

## 13.4 콜드스타트 전략

새 플랫폼이 망하는 이유는 빈 기능을 보여주기 때문이다. 전국에 얇게 깔지 않고 동네에 집중해 그 동네만큼은 꽉 차 보이게 만든다.

- 동네 집중 — 거점 동네(분당/일산/세종 등 맘카페 활성지역) 우선 론칭

- 밀도 미달 시 기능 숨김 — 행정동 기반 밀도 임계값 미달 시 크루/마켓 자체를 숨김(원격 피처 플래그)

- '초대하면 빨리 열려요' — 밀도 미달 동네에서 친구 초대로 바이럴 + 콜드스타트 동시 해결

- 동네 초기 멤버 한정 혜택 — 영구 뱃지 + Pro 체험 + 매물 수수료 면제

## 13.5 Empty State = 기대감 UI

'우리 동네는 아직 준비 중'을 부정적 공백이 아닌 기대감으로 전환한다. Empty state 디자인이 이 앱 성패의 숨은 키.

| 장치 | 내용 |
| --- | --- |
| 진행 바 | '우리 동네 78% 모임 준비됨' |
| 대기 명단 | 오픈 알림 신청 |
| 초대 유도 | '친구를 초대하면 더 빨리 열려요' |

# 기능 14. 데이터·프라이버시 & 지표

아동 데이터는 민감정보다. '데이터 비매각 + 무료 영구 보존'을 신뢰 자산이자 마케팅 무기로 확정한다. R10·R25 토론 반영.

## 14.1 프라이버시 원칙 (절대)

- 사진은 서버에 올라가지 않음 — 로컬/iCloud 우선 저장(무료), 서버 백업은 Pro 혜택

- 데이터 절대 판매 안 함 — 이를 대놓고 마케팅에 활용

- 무료 데이터 영구 보존 — '데이터 인질극' 금지(무료라고 삭제 안 함)

- 데이터 주권 — 언제든 표준 포맷으로 내보내기 가능

## 14.2 또래 비교 (안심 톤)

성장 비교는 '부모 불안 산업'이 되기 쉽다. 백분위 숫자를 크게 보여주면 경쟁심·불안을 유발하므로, 기본은 정성적 안심 메시지, 정밀 수치는 원하는 사람만 펼쳐본다.

| 항목 | 처리 |
| --- | --- |
| 기본 노출 | '또래와 비슷하게 잘 크고 있어요' 정성적 안심 |
| 정밀 수치 | 옵트인 확장(등수 매기기 금지) |
| 집계 방식 | 익명 집계만, 데이터 사용처 설정에서 고지 |
| 최소 표본 | n 임계값(예: 100) 미달 시 비교 비활성(역추적 방지) |

## 14.3 데이터 비매각의 전략적 함의

데이터를 팔지 않는다는 것은 데이터 비즈니스를 포기한다는 의미다. 따라서 수익은 순수하게 구독 + 거래 수수료로 한정한다. 이는 신뢰를 택한 전략적 결단이며, 차별화이자 강력한 마케팅 메시지가 된다.

## 14.4 핵심 지표 체계

허영 지표(다운로드 수)를 버리고 습관·신뢰 중심으로 측정한다.

| 지표 분류 | 내용 |
| --- | --- |
| 북극성 지표 | WAU 중 주 3회 이상 기록/방문 활성 비율 |
| 저빈도 고가치 | 접종·지원금·응급만 쓰는 유저 별도 세그먼트(이들도 가치 유저) |
| 기록 깊이 | 유저당 누적 기록 수, 데이터 풍부도(애착 지표) |
| 마켓·크루 | 성사율 중심(등록 대비 거래완료, 매칭 대비 만남) |
| 빠른기록 완료율 | FAB→저장 완료 비율(가장 중요한 1mm) |
| 기술 건전성 | 크래시율, ANR(앱이 죽으면 다른 지표 무의미) |

## 14.5 분석 방침

- 프라이버시 보전 분석 — 익명·온디바이스 집계, 아동 데이터 외부 유출 금지

- 정량 + 정성 병행 — 이탈 사유는 숫자가 아닌 인터뷰·리뷰·사용성 테스트로

- 긴 시간 지평 — 장기 가치(2년 뒤 아이 앨범) 기능을 단기 지표로 과소평가하지 않음

# 부록 A. 출시 플레이북

첫 2주가 운명을 가른다. '그냥 앱스토어 올리기'가 아니라 거점 집중 → 검증 → 확장 전략. R28 토론 반영.

- 거점 동네 1곳 집중 론칭(분당/일산/세종 등 맘카페 활성지역)

- 서버 부하 점진 증가로 안정적 운영

- 소프트 런칭 동안 온보딩·첫 기록 경험 집중 관찰·개선

## A.2 맘카페 가치 기여형 침투

대놓고 홍보 글은 삭제된다. 진짜 도움 되는 정보·후기로 자연스럽게 스며든다. 지역 맘카페 운영진·파워 회원에게 베타 제안, 1명이 글을 올리면 수백 명이 자연 유입.

## A.3 베타 테스터 커뮤니티

얼리어답터 피드백을 빠르게 반영하고 그 과정을 가시화하면 충성 유저가 된다. 피드백이 실제 제품에 반영되는 것이 보이게 운영.

## A.4 추천 기반 성장

실제 쓰는 양육자의 추천이 유일하게 통하는 마케팅. 광고비보다 입소문(레퍼럴)에 투자.

## A.5 기술 대응 체계

| 항목 | 내용 |
| --- | --- |
| 배포 파이프라인 | TestFlight + CI/CD로 빠른 배포 |
| 심사 대응 | 앱스토어 심사 1~3일 감안한 핫픽스 체계 |

# 부록 B. 기반 아키텍처 & 절대 원칙

초기 아키텍처 결정이 5년을 좌우한다. 기능 범위는 넓게 가되, 1인 개발이 지속가능하도록 확장 가능한 기반을 설계한다. R12·R19·R20 토론 반영.

## B.1 필수 기반 아키텍처

| 요소 | 목적 |
| --- | --- |
| 공통 이벤트 버스 | 기능 간 연결(이정표 달성 → 추천 트리거)을 처음부터 표준화, v2 대공사 방지 |

## B.2 기능 간 연결 (핵심 해자)

당근은 기록이 없고, 기록 앱은 마켓이 없다. '이유식 시작 기록 → 의자 추천 → 동네 양육자 판매 → 크루로 만남'이 한 앱에서 도는 것은 아무도 못 따라온다. 단 강요는 금물 — 유저 행동 직후의 맥락 추천만 허용(이정표 직후, 검색 직후), 상시 푸시 금지. 정보가 주인공, 매물은 조연(콘텐츠 커머스).

## B.3 기술 스택

| 영역 | 스택 |
| --- | --- |
| 클라이언트(MVP) | SwiftUI (속도 우선) |
| 크로스플랫폼(검증 후) | KMP 또는 Flutter 재설계(비용 사전 인지) |
| 로컬/동기화 | CoreData + CloudKit(가족 공유) |
| 백엔드(마켓 이후) | Supabase (Postgres + Auth + Storage + Realtime) |
| 결제 | StoreKit 2(영수증 검증·복원·환불·graceful degradation) |
| AI | Core ML(온디바이스) + 서버 LLM(Pro) |
| 위젯/워치 | WidgetKit + Apple Watch 빠른 기록 |

## B.4 안드로이드 & 확장

- MVP는 iOS(SwiftUI)로 빠르게 검증

- 네트워크 기능(크루/마켓)은 플랫폼 무관이 중요 → 안드로이드 v2~v3 진출 확정

- 검증 후 크로스플랫폼 재설계, 전면 재개발 비용 사전 인지

## B.5 엣지 케이스 정책

소수지만 브랜드 신뢰의 핵심. 무신경한 알림 하나가 SNS에서 치명적으로 퍼질 수 있다. 세심한 배려는 '이 앱 진짜 따뜻하다'는 입소문이 된다.

| 상황 | 정책 |
| --- | --- |
| 상실·휴면 | '기록 멈춤' 모드 — 모든 권유 알림 중단, 미사용 신호로 자동 완화 |
| 이혼·별거 | 가족 공유 해제 시 기존 데이터(기기 저장분) 회수 불가 명시 |
| 다양한 가족 | 입양·위탁·조손가정 포용 — '엄마/아빠' 대신 '양육자/보호자' 언어 |
| 성별 중립 | '○○님' 기본, 맘/파파 선택, 중립 뱃지(육아 동지 등) |

## B.6 절대 원칙 (타협 불가)

- 데이터 비매각

- 무료 데이터 영구 보존

- 무광고 (럭셔리 포지셔닝, 직접 제휴만 허용)

- 정직한 결제 (다크패턴 금지)

- 성별 중립

- 아동 안전

- 안정성 우선 (새 기능보다 버그 없는 경험)

- 데이터 주권 (상시 내보내기)

# 부록 C. 장기 비전 (3년 로드맵)

단순 앱이 아니라 육아 라이프스타일 플랫폼, 나아가 지역 육아 인프라가 되는 것이 목표. 단 데이터 비매각 원칙은 끝까지 유지. R29 토론 반영.

## C.1 3년 비전

지역 육아 인프라 플랫폼 — 기록·거래·커뮤니티·정보가 통합된 동네 육아 허브. 당근이 중고거래에서 지역 생활 플랫폼이 된 것처럼, 육아 버티컬의 통합 허브를 지향한다.

## C.2 생애주기 동반

'아이 크면 떠나는 앱' 문제를 해결하기 위해 생애 단계에 따라 진화한다. 핵심(기록·동네)은 유지하고 생애 단계별 모듈을 플러그인처럼 붙이는 아키텍처로 지속가능성 확보.

| 단계 | 확장 모듈 |
| --- | --- |
| 영유아(0-2세) | 수유·수면·기록, 예방접종, 응급 |
| 유아(3-5세) | 키즈카페·놀이, 어린이집 정보 |
| 학령기(6세+) | 학습·교육, 학원 정보(플러그인 모듈) |

## C.3 부모 정체성 락인

첫째가 커도 둘째가 오고, 육아 경험은 평생 정체성이다. '졸업'이 아니라 '선배맘/선배 양육자'로 전환해 후배 도움·노하우 공유로 잔류시킨다.

## C.4 자녀 데이터 이양 (20년 락인)

아이가 성장하면 기록을 자녀에게 물려준다. '네가 처음 걸은 날이야'라는 평생 가치. 회사가 망해도 데이터는 남도록 표준 포맷 익스포트로 영속성 보장. 20년 데이터는 정서적으로 강력한 차별화이자 장기 락인.

## C.5 회사화 단계

| 시점 | 단계 |
| --- | --- |
| 0~6개월 | 솔로 출시 + 거점 검증 |
| 6~12개월 | 마켓·구독 수익화 |
| 12~24개월 | 법인 설립 + 팀 빌딩 |
| 24개월+ | 플랫폼화 + 안드로이드 |

## C.6 성장해도 지키는 초심

회사가 커진다고 광고를 덕지덕지 붙이거나 과금을 심하게 하면 떠난다. 성장하면서도 '가볍고 신뢰할 수 있는' 초심을 유지하는 것이 장기 생존의 조건. 광고·과금 절제, 신뢰 우선 원칙 고수.

## C.7 30라운드 종합 결론

이 앱의 승부처는 기능 개수가 아니라 세 가지다. 첫째, 매일 닿는 1mm(기록·홈)의 완성도. 둘째, 못 베끼는 해자(통합·동네 밀도·데이터 축적에 따른 switching cost). 셋째, 신뢰(프라이버시 비매각·정직·무광고). 대기업과 정면승부를 피하고, 뾰족한 공감과 따뜻한 디테일로 팬덤을 만들고, 데이터 축적으로 잠근다. 좁게 시작하더라도 기반은 넓게, 가볍고 깊고 따뜻하게.
