# CLAUDE.md — BabyLog

BabyLog는 임신부터 육아까지 아우르는 iOS 육아 슈퍼앱이다. 1인 개발 프로젝트이며, 전체 기능 명세는 `SPEC.md`에 있다. 작업 전 반드시 `SPEC.md`의 해당 기능 섹션을 먼저 읽는다.

포지셔닝: "우리 동네 육아의 모든 것" — 아이의 성장을 평생 담고(기록), 동네 이웃과 잇는(거래·크루) 단 하나의 앱.

## 프로젝트 개요

플랫폼은 iOS 우선이며 SwiftUI로 개발한다. 검증 후 안드로이드(KMP 또는 Flutter)로 확장할 계획이므로, 비즈니스 로직과 UI를 분리해 이식성을 확보한다. 로컬 저장은 CoreData, 가족 동기화는 CloudKit을 쓴다. 마켓 등 네트워크 기능의 백엔드는 Supabase(Postgres + Auth + Storage + Realtime), 결제는 StoreKit 2를 사용한다.

기능은 총 14개로 임신 기록, 성장 기록, 주변 인프라/응급, 중고 마켓, 가계부, 동네 크루, 뱃지, 온보딩, 홈/리텐션, 알림, AI, 접근성, 신뢰·안전, 데이터·프라이버시로 구성된다. 상세는 `SPEC.md`를 참조한다.

## 절대 원칙 (타협 불가)

이 원칙들은 모든 구현에서 예외 없이 지킨다.

- 데이터 비매각 — 아동 데이터는 절대 외부에 판매하지 않는다. 수익은 구독과 거래 수수료로만.
- 무료 데이터 영구 보존 — 무료 사용자의 데이터도 삭제하지 않는다. "데이터 인질극" 금지.
- 사진은 서버에 올리지 않는다 — 무료는 로컬/iCloud 저장, 서버 백업은 Pro 혜택. 이것을 마케팅 포인트로 활용한다.
- 무광고 — 광고 SDK를 도입하지 않는다. 럭셔리 포지셔닝과 상충. 직접 제휴만 허용.
- 정직한 결제 — 다크패턴 금지. 자동결제 사전 고지, 해지는 쉽고 존중하는 톤.
- 성별 중립 — UI 전반에서 '○○맘' 대신 '양육자/○○님'을 기본으로. 맘/파파는 선택. 아빠·조부모·다양한 가족(입양·위탁·조손) 포용.
- 아동 안전 — 아이 관련 기능에서 안전은 최우선. 사고 한 건이 브랜드를 죽인다.
- 안정성 우선 — 새 기능보다 버그 없는 경험. 자주 튕기는 앱은 모든 가치가 무의미.
- 데이터 주권 — 사용자가 언제든 표준 포맷으로 데이터를 내보낼 수 있어야 한다.

## 민감 영역 (특별 주의)

임신 기록과 관련해 유산·사산 등 상실 가능성이 있는 가장 민감한 영역이다. 다음을 반드시 지킨다.

- "기록 멈춤" 모드 — 상실 시 모든 주차 알림·태아 가이드·권유 알림을 즉시 중단할 수 있어야 한다.
- 미사용 신호 자동 감지 — 오래 미접속 시 권유 알림을 자동 완화한다. 사용자가 직접 설정하는 부담을 주지 않는다.
- 카피는 따뜻하고 중립적으로 — 상실 가능성을 염두에 둔 표현. 절대 닦달하지 않는다.

의료 조언은 직접 제공하지 않는다. 증상 관련 질의는 "병원 방문 권장 + 주변 소아과 연결"로 대응하고, 모든 태아·건강 정보에는 "의료 상담을 대체하지 않음" 면책을 명시한다.

## 아키텍처 규칙

초기 아키텍처 결정이 5년을 좌우한다. 다음 기반을 처음부터 설계한다.

- SPM 모듈화 — 기능별로 독립 Swift Package로 분리해 1인 유지보수 부담을 분산한다.
- 공통 이벤트 버스 — 기능 간 연결(예: 이정표 달성 → 매물/정보 추천 트리거)을 처음부터 표준화한다. 이걸 안 하면 나중에 대규모 리팩터링이 발생한다.
- 다국어 문자열 분리 — 텍스트를 하드코딩하지 않고 문자열 리소스로 분리한다. 미래 국제화 옵션을 비용 없이 확보.
- 데이터 표준 익스포트 — 데이터 주권·자녀 이양·서비스 종료에 대비한 표준 포맷 내보내기를 지원한다.
- 원격 구성(피처 플래그) — 동네별 기능 점진 개방, 심사 없는 핫픽스를 위해 서버 제어 구성을 둔다.
- 온디바이스 우선 — 데이터 처리는 가능한 한 온디바이스로. 프라이버시 보전 + 서버 비용 절감.
- AI 비용 분기 — 온디바이스(Core ML) 가능한 기능은 무료, 서버 LLM 필요한 기능은 Pro. 'AI' 단어를 전면에 내세우지 않고 매끄러운 경험으로 내재화.

## 핵심 데이터 전환

임신 기록(Pregnancy)은 출산 시점에 아이(Child)로 승계된다. 배 사진 타임라인이 성장 사진으로, 태명이 아이 이름으로, 예정일이 실제 생년월일로 이어진다. 태아 시절부터 성장까지 끊김 없는 하나의 여정을 형성하는 것이 핵심 가치이므로, 데이터 모델 설계 시 이 연속성을 반드시 고려한다.

## UX 원칙

- 기록 입력은 2탭 완료 — FAB 탭 → 사진 선택 → 저장. 사진 1장이면 기록 완료. 강제 입력 단계는 0.
- 한 손 조작 — 중요 버튼은 모두 엄지가 닿는 하단 중심에 배치. 아이를 안고 쓰는 환경.
- 접근성 내재화 — Dynamic Type, VoiceOver, 색약 대응(색+아이콘+레이블 3중 인코딩)을 처음부터. 나중에 소급하면 전면 재작업.
- 또래 비교는 안심 톤 — 백분위를 등수 경쟁처럼 보여주지 않는다. 기본은 정성적 안심 메시지, 정밀 수치는 옵트인.

## 기술 스택 요약

| 영역 | 스택 |
| --- | --- |
| 클라이언트 | SwiftUI (iOS 16+) |
| 로컬/동기화 | CoreData + CloudKit |
| 백엔드 | Supabase (Postgres, Auth, Storage, Realtime) |
| 결제 | StoreKit 2 |
| AI | Core ML (온디바이스) + 서버 LLM (Pro) |
| 차트 | Swift Charts |
| 위젯/워치 | WidgetKit + Apple Watch |
| 카드 합성 | ImageRenderer + Core Graphics |

## 외부 API (전부 무료)

- 질병관리청 예방접종도우미 — 예방접종 스케줄
- 건강보험심사평가원 — 소아과/약국 정보
- 카카오맵 로컬 API — 주변 장소(키즈카페 등)
- 복지로 API — 정부지원금(아동수당·부모급여·첫만남이용권 등)
- KATSA/KERI 리콜 DB — 카시트 등 리콜 조회

## 작업 방식

기능을 구현할 때는 `SPEC.md`에서 해당 기능 섹션(예: "기능 1. 임신 기록")을 먼저 읽고, 데이터 구조·기술 스택·성공 지표를 확인한 뒤 시작한다. 새 코드는 위 아키텍처 규칙(SPM 모듈화, 이벤트 버스, 문자열 분리)을 따른다. 절대 원칙과 민감 영역 규칙은 어떤 경우에도 우회하지 않는다.
